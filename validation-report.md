## Публичная поверхность: без бренда инструмента («Public surface contains no Lumen»)

Перед публикацией пакета на продакшен-URL продукта проверить, что **потребительские** артефакты и метаданные **не** содержат имя локального инструмента (**Lumen**) там, где это публичная выдача (см. также `docs/spec/lumen-product-spec.md`, критерии validation/lint).

Чеклист (ручной или в CI после сборки сайта):

- [ ] **`llms.txt`** — нет подстроки `Lumen`, нет путей вида `/lumen/`, нет формулировок «Lumen package» / «Lumen API»; ссылки на канонические документы ведут на **`/product-data/docs/*.md`**, не на **`/knowledge/*`** (MVP).
- [ ] **Публичные URL** JSON/JSONL/манифестов — нейтральный префикс **`/product-data/`** (без **`/lumen/`**).
- [ ] **Канонические Markdown-документы** опубликованы под **`/product-data/docs/*.md`** (`text/markdown`); префикс **`/knowledge/*`** в MVP **не** используется как канон или настроен как HTML-зеркало с **`alternate`** / без конфликта canonical.
- [ ] **`sitemap.xml`** — только нейтральные канонические URL пакета; нет обязательных ссылок на инструмент как продукт.
- [ ] **Canonical и SEO metadata** (HTML-хост) — описания и `canonical` без бренда инструмента упаковки.
- [ ] **Тела JSON / JSONL** (включая примеры на сайте, если есть) — нет `"Lumen"` как торговой марки пакета в пользовательских полях; публичные примеры URL не используют `lumen.local`, `/lumen/`, `api.example.com` как «официальный» продакшен-хост.
- [ ] **JSON Schema `$id`** в артефактах пакета — нейтральный URN (`urn:product-knowledge-package:schema:…`), без `lumen.local`.

Допустимы **внутренние** упоминания Lumen в репозитории (README для разработчиков, правила агентов, спецификация о локальном инструменте) — этот чек относится к **опубликованному** слою.

## Интеграция со сторонним проектом

Конспект см. **`docs/integration/third-party-product-data.md`**. Перед включением маршрутов в прод:

- [ ] Скопирован/синхронизирован выход **`generated/product-data/`** из репозитория упаковки в артефакт сборки сайта (`public/` или путь Route Handler).
- [ ] Живые URL **`/llms.txt`**, **`/product-data/*`** (в т. ч. **`/product-data/docs/*.md`**), **`sitemap`** перечисляет пакет и канонические документы, **`robots`** содержит **`Sitemap:`**.
- [ ] Корректные **`Content-Type`** для JSON, NDJSON, plain text Markdown, см. таблицу в интеграционном документе.
- [ ] **Утечки локальных путей:** отсутствуют типичные сигнатуры машины разработчика (**`/Users/`**, **`Developer/Projects`**, сегмент **`workspace/`** в путях) в тексте публикуемых артефактов; см. **`npm run validate:package`** (`public_local_path_*`).
- [ ] Build/test pipeline стороннего проекта проходит после подключения package.

## Claims ↔ Evidence

| `claim_id` | Утверждение | `truth_status` (evidenced / …) | Связанные `evidence_id` | Пробелы / риски |
|------------|-------------|---------------------------|--------------------------|----------------|

#### Missing evidence

- Для утверждений не в статусах hypothesis / policy_forbidden: есть доказательства в модели и/или `claim_supported_by` в entities?
- [ ] Нет утверждений «как проверенных», которые не имеют evidence и не помечены иначе.

#### Broken relations

- Каждый `target_id` в `relations` существует в `entities.jsonl` (или в явно разрешённом внешнем ref).
- [ ] Нет недостижимых `target_id` в графе сущностей.

#### Orphan entities

- Ключевые сущности достижимы от узла продукта по графу (или документированы как допустимый leaf без продукта).
- [ ] Список сирот пуст или зафиксирован как исключение.

| `entity_id` | Тип | Комментарий |
|-------------|-----|-------------|
|             |     | |

#### Устаревшие утверждения (stale claims)

- Версии/даты claims согласованы с актуальностью связанных evidence и текста канонических документов / чанков.
- [ ] Нет утверждений новее evidence противоречием по факту среды описания.

#### Duplicate claims

- Нет конфликтующих дублей одной и той же нормализованной истины без различия scope или id.
- [ ] Дубли устранены или разведены явным ограничением применимости.
