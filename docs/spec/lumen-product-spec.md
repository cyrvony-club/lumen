# Lumen — продуктовая спецификация

Единый источник истины по **локальному инструменту Lumen** и по форме **публикуемого нейтрального пакета данных о продукте**. Агентный workflow разработки репозитория в пользовательский сценарий **не входит** (см. `.cursor/rules/lumen-agent-system.mdc`).

---

## Локальный инструмент и публичный output

| Слой | Что это |
|------|---------|
| **Локальный инструмент (Lumen)** | Внутренняя сборка/CLI/пайплайн в этом репозитории; имя допустимо во внутренней документации и коде инструмента. |
| **Публичный output** | **Product knowledge package** / **LLM-readable product data package** / **machine-readable product data** — отдаётся на сайте продукта **без бренда Lumen**: не использовать в публичных URL, `llms.txt`, SEO metadata и открытых файлах формулировки «Lumen package», префиксы `/lumen/*`, «Lumen API/endpoints». |

---

## Назначение Lumen

**Lumen — локальный инструмент упаковки продуктовой правды** в публикуемый **product knowledge package** (**LLM-readable product data package**), чтобы внешние LLM, RAG и агентные системы могли **корректно понять, извлечь и упомянуть** продукт.

Публичный пакет задаёт канонические факты, структуру знания (сущности, связи, чанки) и отчёт о проверке согласованности — без обещания «попасть в ответ» конкретной модели или «взломать» ранжирование.

---

## Проблема

Публичные **сайты и разрозненные материалы** плохо подходят как источник знания для LLM:

- текст рассчитан на человека, а не на стабильное извлечение фактов;
- нет явной связи «утверждение → доказательство»;
- дублирование и противоречия не контролируются;
- нет машинного «оглавления продукта» для retrieval.

Цель упаковки — снизить искажение и пробелы при последующих запросах к моделям и при RAG-ingest **стороними** системами.

---

## Продуктовая гипотеза

**Если** продуктовую правду представить как **структурированный, проверяемый и машинно-читаемый** пакет (модель правды → канонические документы → сущности и чанки → фиды),

**то** LLM / RAG / агентные системы смогут **лучше распознать продукт**, оценить релевантность и использовать **подкреплённые фактами** формулировки в ответах.

Гипотеза **не** утверждает гарантию цитирования бренда в ответах ChatGPT / Claude / Gemini.

---

## Что Lumen делает

- Принимает **входы** владельца продукта (бриф, позиционирование, утверждения, доказательства, контекст конкурентов, целевые запросы, ограничения).
- Строит **product truth model** — нормализованное описание продукта и утверждений со связями на evidence и политики.
- Генерирует **канонические LLM-readable документы** (Markdown): обзор, сценарии, сравнения, FAQ, доказательная база — **человекочитаемые заголовки и списки** с явной трассируемостью полей модели (**`claim_id`**, **`evidence_id`**, **`truth_status`**, при наличии **`source_refs`**), плюс **verbatim JSON-фрагменты только из входной** `product-truth-model`; **новые утверждения или маркетинг генератор не добавляет**.
- Публикует **сущности** (`entities.jsonl`) и **чанки** (`chunks.jsonl`) для ingest и retrieval во внешних системах.
- Ведёт **манифест обновлений** (`updates.json`) для инкрементальной синхронизации.
- Формирует **validation report** — явная проверка полноты, противоречий и трассируемости claim → evidence.

## Что Lumen не делает

- Не ставит **написание постов в соцсетях или маркетинговый копирайт** основной функцией продукта.
- Не гарантирует **появление продукта** в ответах конкретных LLM.
- Не **манипулирует ранжированием** моделей или «GEO/LLM SEO» как обещание результата.
- **Не является агентной системой как продукт** — только пакеты знаний и артефакты вокруг продуктовой правды.
- **Не ставит открытый HTTP API в центр MVP** без доказанной потребности; контракт API — опциональное расширение (см. `api/openapi.yaml`).
- Не допускает **выдуманных преимуществ** без указания evidence или статуса гипотезы в модели правды и отчёте валидации.

---

## Пользователь и сценарий

**Пользователь:** владелец продукта / PM / growth / founder, который хочет, чтобы модели и RAG **опирались на согласованные факты**, а не на случайный веб-слой.

**Сценарий (целевой):**

1. Заполнить входы (бриф, позиционирование, утверждения, доказательства, конкуренты, целевые запросы, ограничения).
2. Собрать **product truth model** и утвердить спорные утверждения (или пометить как «гипотеза» с низкой уверенностью).
3. Сгенерировать **пять канонических документов** и JSONL экспорты.
4. Передать пакет во внешний RAG / агент **как корпус** или через опциональный retrieval API.
5. Прочитать **validation report** и устранить блокеры перед публикацией новой версии пакета.

## Локальная сборка пакета в репозитории

Команда **`npm run generate:package`** (см. `README.md`) читает внешний или временный локальный `product-truth-model.json` через **`--input`** и записывает **нейтральный** комплект файлов по умолчанию в **`generated/product-data/`** (`index.json`, `llms.txt`, копия truth model, `docs/*.md`, `entities.jsonl`, `chunks.jsonl`, `updates.json`, `sitemap.xml`, `graph.json`). Корневой `product-truth-model.json` допустим только как временный рабочий input и **не должен становиться знанием Lumen о конкретном проекте**. Публичные URL задаются параметром **`--base-url`** (по умолчанию документирован как mock `https://example.com/product-data` в CLI). **`generated/`** исключена из версионирования. После сборки общий пайплайн **`npm run validate`** при наличии сгенерированного набора выполняет для него дополнительные те же классы проверок, что и для примеров под `exports/`.

## Входные данные

| Вход | Содержание |
|------|------------|
| **Product brief** | Название, категория, сайт, краткое описание, целевая аудитория |
| **Positioning** | Use cases; кому подходит / **не** подходит |
| **Claims** | Утверждения о продукте (в т.ч. границы применимости) |
| **Evidence** | Ссылки, цитаты, метрики, скриншоты, исследования — привязка к claims |
| **Competitor context** | С кем сравнивают, отличия, таблицы (без недоказанных нападок) |
| **Target LLM queries** | 5–8 формулировок запросов, под которые пакет должен давать извлекаемые ответы |
| **Source of truth** | Что считается каноном при конфликте (док, раздел, владелец) |
| **Policy constraints** | Юридические формулировки, запреты, дисклеймеры, PII, региональные ограничения |

## Выходные артефакты

| Артефакт | Назначение |
|----------|------------|
| `llms.txt` | Компактный указатель канонических URL пакета для LLM-агентов и внешних систем; тексты и ссылки **без упоминания Lumen** |
| `product-truth-model.json` | Машинно-читаемая модель правды (см. схему `schema/product-truth-model.schema.json`) |
| **Canonical docs** (Markdown) | `product-overview.md`, `use-cases.md`, `comparison-context.md`, `faq.md`, `evidence.md` — читаемый Markdown + трассируемые идентификаторы и статусы; публичный канон MVP — **`/product-data/docs/<имя-файла>`**; факты строго из входной модели правды |
| `entities.jsonl` | Сущности продуктовой видимости и связи |
| `chunks.jsonl` | Фрагменты для ingest и retrieval |
| `updates.json` | Манифест изменений версии пакета (`feeds/updates.json`; при сборке рядом с сайтом — публичный URL) |
| `sitemap.xml` | Карта стабильных URL для поискового индекса и синхронизации потребителей |
| Производный граф *(опционально)* | например `exports/graph.json`: узлы (`nodes`) и рёбра (`edges`) для обходов и аудита связей |
| `validation-report.md` | Итоги проверок согласованности и покрытия |
| `index.json` | **Генерируемый** манифест сборки пакета: версия, время генерации, `product_id`, список артефактов с публичными URL и MIME (см. ниже) |
| HTTP API | **Опциональное расширение, не блокирует MVP**; черновик возможных retrieval-endpoints см. `api/openapi.yaml` (**локальный/mock**, без имени Lumen на публичной поверхности сайта) |

### index.json (генерируемый манифест пакета)

Файл выдаётся как часть **публичного** product knowledge package (типичный путь — `/product-data/index.json`). Минимальный контракт:

| Поле | Тип | Назначение |
|------|-----|------------|
| `package_version` | string | Версия сборки пакета |
| `generated_at` | string (ISO-8601 date-time) | Время генерации манифеста |
| `product_id` | string | Идентификатор продукта в пакете |
| `artifacts` | array | Записи об артефактах сборки |

Элемент `artifacts[]`:

| Поле | Тип | Назначение |
|------|-----|------------|
| `type` | string | Логический тип (`product-truth-model`, `entities-jsonl`, `chunks-jsonl`, `updates`, `graph`, `llms-txt`, `canonical-doc`, …) |
| `url` | string (URI) | Публичный URL артефакта |
| `media_type` | string | MIME (`application/json`, `application/x-ndjson`, …) |
| `checksum` | string (optional) | Контрольная сумма (например SHA-256) для верификации |

### Product knowledge package (форма публичного результата)

**Единицей поставки** считается **целый публичный product knowledge package** из артефактов выше — не отдельный Markdown и не только JSONL без модели правды. Пакет:

- даёт LLM/RAG однозначные **canonical** источники (модель + канонические тексты + манифест обновлений);
- сохраняет **трассируемость** «утверждение → источник/доказательство» там, где требуются факты о продукте;
- поддерживает **инкрементальное потребление** через `updates.json` и счётчики версии сборки уровня пакета;
- имеет машинную **валидацию** качества («validation report», проверки схем экспорта, связности id).

Подробнее для входа разработчиков и агентов — `README.md` в корне репозитория.

## Product truth model

JSON-документ, который:

- идентифицирует продукт (`product_id`, метадата, версия схемы);
- содержит **нормализованные** утверждения (claims) с типом (напр. возможность, ограничение, метрика) и **`truth_status`** (`evidenced` / `hypothesis` / `policy_forbidden`);
- для `evidenced` требует непустой список **`evidence_ids`**; блок **`evidence`** обязателен и не пуст; **`target_llm_queries`** (5–8 непустых `query`), **`source_of_truth`** — обязательные поля;
- связывает утверждение с **evidence** (или явно помечает гипотезу / политический запрет);
- фиксирует **политические ограничения** и **source of truth**;
- хранит **target_llm_queries** для проверки покрытия в validation report.

Цель модели — **single structured source** перед генерацией текста канонических документов и перед нарезкой на чанки.

---

## Canonical LLM-readable docs

Набор из **ровно пяти** Markdown-документов; **публичный MVP-канон** — стабильные URL **`/product-data/docs/<slug>.md`** (slug совпадает с именем файла в пакете). Отдельные HTML-оглавления под другим префиксом (например **`/knowledge/*`**) — **не** часть MVP и допускаются позже только как зеркало/`alternate` без дублирующего canonical.

| Файл | Содержание |
|------|------------|
| `product-overview.md` | Что это, для кого, ключевые возможности без маркетингового шума |
| `use-cases.md` | Сценарии, подходит / не подходит |
| `comparison-context.md` | Рамки сравнения с конкурентами + оговорки |
| `faq.md` | Вопросы и ответы в форме извлекаемых блоков |
| `evidence.md` | Карта «claim → источник / артефакт доказательства» |

Стиль — **ясные заголовки, буллеты фактов, минимум неясных местоимений**, чтобы чанкинг и retrieval давали воспроизводимые отрывки.

## Сущности и связи

Сущности типизированы вокруг **product visibility**: `Product`, `Feature`, `UseCase`, `Audience`, `Category`, `Competitor`, `Claim`, `Evidence`, `FAQ`, `Comparison`.

- Каждая сущность имеет стабильный `id`, `type`, человекочитаемый заголовок (`title` или `name`), краткое `summary`, теги, `updated_at`, `version`, `canonical_url`.
- Опционально **`source_refs`**: указатели на immutable raw источники и канон для трассируемости compiled truth layer (та же форма объекта, что в `schema/chunk.schema.json` и в элементах `evidence[].source_refs` модели правды).
- **Направление связи:** в `relations[]` каждая запись читается как **текущая сущность (subject) —[type]→ `target_id` (object)**. Например, у **`Product`** исходящие **`product_has_feature`** / **`product_has_claim`** указывают на `Feature` / `Claim`; у **`Claim`** исходящее **`claim_supported_by`** указывает на `Evidence`. Ошибочно вешать на `Feature` ребро `product_has_feature` в сторону `Product` (это снова «продукт владеет фичей», но с неверным субъектом).
- Связи (`relations[].type`): зарегистрированный перечень в JSON Schema включает, например, `claim_supported_by`, `feature_serves_use_case`, `product_has_feature`, `compares_to`, `evidences_product` (полный список — см. `schema/entity.schema.json`).
- Для типа **`Claim`**: связь через `claim_supported_by` на `Evidence`, **или** поле **`truth_status: hypothesis`**/**`policy_forbidden`** (согласовано с `truth_status` в `product-truth-model.json`).

Схема строки: `schema/entity.schema.json`.

---

## Chunks

**Chunk** — **LLM-readable фрагмент** текста, предназначенный для ingest во внешний vector-store или keyword-index и последующего retrieval: достаточно самодостаточный абзац или секция, с метаданными привязки к документу и сущностям.

Поля см. `schema/chunk.schema.json`: `chunk_id`, `doc_id`, заголовок, `text`, теги, время обновления; опционально `source_refs`, `entity_ids`, `claim_ids`, `evidence_ids` для связности с моделью правды и сыроценными источниками.

---

## Exports и feeds

- **`exports/entities.jsonl`** — поток объектов Entity (по одной JSON-строке на сущность).
- **`exports/chunks.jsonl`** — поток Chunk для сторонних пайплайнов индексации.
- **`feeds/updates.json`** — манифест изменений (время сборки + версия пакета + счётчики изменений + элементы изменений) для синхронизации потребителей пакета.

Форматы согласованы с тем, что описывает **опциональный** `api/openapi.yaml` там, где пересекаются поля (`/updates`, сущности, чанки). Наличие OpenAPI не означает обязательной реализации HTTP-сервера в MVP.

## Публикация и доступ LLM

Публичный **product knowledge package** должен быть доступен **публичными URL без аутентификации**. Не использовать как единственный канал публикации режим «только за логином», «только исполняемый JS без текстов краулерам» или «только закрытый PDF без явного текстового канала».

Типичные точки входа для систем:

- Корневой или продуктовый **`/llms.txt`** — ссылки на ключевые URL пакета (модель, фиды, **канонические Markdown** под **`/product-data/docs/*.md`** как `text/markdown`). **`llms.txt` не именует Lumen** и не ссылается на префиксы `/lumen/*`.
- Стабильный **нейтральный** префикс **`/product-data/*`** для JSON/JSONL и канонических документов, например **`/product-data/index.json`**, **`/product-data/entities.jsonl`**, **`/product-data/chunks.jsonl`**, **`/product-data/updates.json`**, **`/product-data/docs/product-overview.md`**, … при необходимости — производного графа. Префикс **`/knowledge/*`** для HTML-зеркал **не** входит в MVP и вводится только без конфликтующего второго canonical (см. [`docs/integration/third-party-product-data.md`](../integration/third-party-product-data.md)).
- **`/sitemap.xml`**, **`/robots.txt`** или программный `robots.ts` — явное разрешение индексирования путей с артефактами пакета; **canonical URLs** в HTML-хостах согласованы с теми же стабильными путями.
- Отдача **JSON и JSONL** с корректным `Content-Type` (`application/json`, `application/x-ndjson` или `application/jsonl` при принятии в инфраструктуре потребителя).
- Структурированное потребление **без своего сервера** — search crawl + индексирование JSONL экспортов клиентским retrieval.
- **Optional retrieval/search HTTP API** (см. `api/openapi.yaml`) — черновик **локального/mock** контракта (`servers.url` ориентирован на `http://localhost:3000`), не обязательная публичная часть MVP; публичная поверхность сайта **не обязана** именовать или раскрывать инструмент. Для минимальной поставки HTTP-сервер **не обязателен** при сценарии «публичный сайт + статические артефакты» — **первую реализацию кода в этом репозитории начинать с валидаторов и генерации пакета** (`validate:*`, сбор `index.json`, JSONL, модели правды), а не с подъёмом retrieval-сервера.

## Интеграция с индексируемым сайтом (Next.js)

Цель — чтобы **полный текстовый и структурный слой** пакета жил там же, где продуктовый сайт, и был машинно обнаружим. Конкретные пути задаёт хостинг; описание ниже — **ориентир реализации** на базе типового **Next.js App Router** (не жёсткое требование к репозиторию инструмента Lumen).

Рекомендуемые строительные блоки интеграции:

| Элемент Next.js App Router | Роль для публичного пакета |
|----------------------------|----------------------------|
| `public/llms.txt` (или сборка того же содержания) | Статический указатель входа для LLM-агентов (**без имени Lumen**) |
| `public/robots.txt` **или** `src/app/robots.ts` | Разрешения/пути для краулера к нейтральному префиксу (напр. `/product-data`) и каноническим документам |
| `src/app/sitemap.ts` (`sitemap.xml`) | Перечень URL публичных артефактов и ключевых страниц продукта |
| `src/app/product-data/[...segments]/route.ts` | Отдача JSON/JSONL/manifest и **`docs/*.md`** из `Route Handler` со стабильными URL вида `/product-data/index.json`, `/product-data/entities.jsonl`, **`/product-data/docs/<slug>.md`** |
| Metadata / layouts | `canonical` и описания там, где артефакты отдаются как HTML-страницы (**SEO без бренда инструмента**) |
| `Content-Type`, кэш | Предсказуемые типы тел ответов; кэширование версионированных путей через `immutable` там, где уместно |

Lumen не хранит знания о конкретных проектах, для которых создаётся package. Интеграция всегда описывается как **контракт для стороннего проекта**: Lumen генерирует нейтральный набор файлов, а внешний сайт, CDN или backend самостоятельно решает, как опубликовать `/llms.txt`, `/product-data/*`, sitemap и robots. Универсальная спецификация размещения: [`docs/integration/third-party-product-data.md`](../integration/third-party-product-data.md).

## Модель обработки данных (data processing)

Входы и промежуточные артефакты следует вести с дисциплиной подготовки **compiled truth layer** над **immutable raw sources** (по аналогии с подходами knowledge/wiki-пайплайнов):

| Слой | Содержание |
|------|------------|
| **Immutable raw sources** | Неизменяемые копии брифа, экспортов интервью, URL-снимков, приложенных документов с фиксацией времени/версии. |
| **Compiled truth layer** | Нормализованный `product-truth-model.json`, канонические Markdown, утверждённые entity/chunk экспорты. |
| **`source_refs`** на сущностях и в чанках | Указание на сыроц или канонический первоисточник (uri, метка, при необходимости время выборки); трассируемость при спорных правках. |
| **Граф и экспорт** | Производные представления (напр. `exports/graph.json`): узлы продуктовой модели visibility и рёбра типизированных связей. |
| **Updates manifest** | `updates.json`: версия пакета, время генерации, счётчики и списки изменённых id — для синхронных обновлений индексов. |
| **Validation / lint** | Схемы JSON Schema Draft 2020-12 (`$id` — нейтральный `urn:product-knowledge-package:schema:*`), построчная проверка JSONL, содержательные проверки в `validation-report.md` (полнота evidence, orphans, противоречия), **чек публичной поверхности** (раздел *Public surface contains no Lumen* в `validation-report.md`: `llms.txt`, URL, sitemap/canonical/metadata, JSON/JSONL, `$id` схем), **`validate:package`** — также отсутствие типичных локальных файловых путей машины разработчика (`/Users/`, `workspace/`, `Developer/Projects`) в сканируемых артефактах и строках JSONL. |

Цель модели — **не смешивать** «сырую цитату» и нормализованный утверждённый факт без явной связи через `source_refs` и элементы модели правды.

## Инициализация реализации (TypeScript / Node.js)

Реализация CLI/валидаторов в этом репозитории стартует с **строгого npm-проекта** (`package.json`, `src/`, см. таблицы ниже).

**Текущий порядок:** сначала **валидаторы и генерация артефактов пакета** (модель правды, `entities.jsonl`, `chunks.jsonl`, `index.json`, `updates.json`, отчёт валидации), затем при необходимости — опциональный HTTP-слой по OpenAPI без позиционирования как необходимость MVP.

Целевой стек кодовой реализации зафиксирован как **TypeScript + Node.js LTS** со **strict TypeScript**.

Активные скрипты `package.json`:

| Script | Цель |
|--------|------|
| `typecheck` | `tsc --noEmit`, strict |
| `lint` | ESLint по исходникам валидатора |
| `test` | Vitest для ключевых проверок связности и публичной поверхности |
| `validate:schema` | компиляция `schema/*.schema.json`; при наличии `product-truth-model.json` — сверка по схеме |
| `validate:exports` | построчная проверка JSONL по `schema/entity.schema.json` и `schema/chunk.schema.json` |
| `validate:package` | синтаксис `feeds/updates.json`, `exports/graph.json`, `api/openapi.yaml`; антиутечки бренда в публичных артефактах; согласованность направлений связей, claim→evidence в чанках и совпадение relations с графом |
| `validate` | запуск трёх валидаторов подряд |

Реализация валидаторов: `src/validators/*`, CLI: `src/cli/validate-*.ts`. Правило Cursor для агентов по стеку: `.cursor/rules/lumen-ts-node.mdc`.

## Validation report

Файл `validation-report.md` (шаблон в корне репозитория) фиксирует:

- проверку **противоречий** между claims и текстом канонических документов;
- **покрытие target LLM queries** (есть ли в пакете извлекаемые ответы без противоречий);
- трассируемость **claim → evidence** и флаги «без доказательств»;
- соблюдение **policy constraints**;
- **публичная поверхность без бренда инструмента** (чеклист *Public surface contains no Lumen* в шаблоне отчёта);
- блокирующие / некритические замечания и версию проверенного пакета.

**Критерий готовности MVP:** по каждому из целевых запросов пакет содержит **точные, извлекаемые и непротиворечивые** факты о продукте (см. раздел MVP).

## Продуктовый pipeline

End-to-end цепочка артефактов:

```text
sources → product truth model → canonical LLM-readable docs → entities → chunks → exports/feeds → validation
```

На этапе **sources** входы собираются и нормализуются; далее каждый шаг производит машинный или текстовый артефакт из таблицы выше.

## MVP

**Объём нормального MVP:**

| Параметр | Значение |
|----------|----------|
| Продуктов в пакете | **1** |
| Target LLM queries | **5–8** |
| Canonical docs | **5** файлов (перечень выше) |
| Артефакты | product truth model, `llms.txt`, `index.json`, `entities.jsonl`, `chunks.jsonl`, `updates.json`, `validation-report.md`, `sitemap.xml` (рядом с сайтом) |
| Публичный generation API | **Нет** (не входит в MVP; не возвращать без решения архитектора) |
| Любой публичный HTTP API (`api/openapi.yaml`) | **Опционально**; mock при необходимости; отсутствие сервера **не блокирует** MVP при публикации статических артефактов |

**Критерий готовности:** по каждому целевому запросу содержится непротиворечивая информация в модели правды и/или канонических документах и она находится в чанках в форме пригодной для извлечения. Минимальная публикация результатов допускает **файловую отдачу** (`llms.txt`, `sitemap`, JSONL) без обязательного backend.

## Future extensions

- Полноценный **retrieval HTTP API** (поиск, чанк, связанные сущности, документы по slug).
- Автоматическая **пересборка** пакета по webhook от source of truth.
- Мультипродуктовые workspace’ы и **версионирование** с миграцией сущностей.
- Расширенные типы evidence (подписанные файлы, внутренние дампы с redaction).

---

## Anti-patterns

- Путать **внутренний агентный workflow** Cursor с тем, что покупает или запускает пользователь Lumen.
- Обещать **гарантированную видимость** в ChatGPT и аналогах.
- Строить MVP вокруг **генерации постов**, а не вокруг упакованной правды.
- Отдавать чанки **без связи с evidence** для ключевых claims.
- Пускать в пакет **непроверяемые сравнения** с конкурентами.
- Центрировать продукт на **API** без сценария потребления пакета файлами/фидами.
