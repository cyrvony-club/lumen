# Пайплайн контента Lumen

## Каноническая цепочка

`brief` → `scout research` → `architect decision` → `worker draft` → `entity extraction` → `chunk export` → `publish` → `validate`

## Этапы (сквозной поток)

1. **Brief** — создаётся `ContentBrief` (цель, аудитория, канал, ограничения, ссылки на `StyleGuide` / `Audience` / `Channel`).
2. **Scout research** — скаут собирает и нормализует `Source`, фиксирует выжимки; при необходимости обновляет каталог источников проекта.
3. **Architect decision** — архитектор утверждает структуру артефакта, шаблон (`ContentTemplate`), промпты (`Prompt`), параметры `GenerationPipeline`, критерии review.
4. **Worker draft** — рабочий выполняет генерацию по утверждённому пайплайну; создаётся `GeneratedArtifact` в статусе `draft` или `generated` (см. жизненный цикл).
5. **Entity extraction** — из текста/метаданных извлекаются или обновляются доменные **сущности** (`Project`, концепты и др.) согласно `schema/entity.schema.json`.
6. **Chunk export** — контент нарезается на чанки с `chunk_id`, заголовком, кратким summary, привязкой к `doc_id` / `entity_ids`; выгрузка в `exports/chunks.jsonl`.
7. **Publish** — объекты получают `canonical_url`, обновляются `feeds/updates.json` и `changelog.md`; статус артефакта — `published`.
8. **Validate** — проверка полноты метаданных, доступности API, согласованности OpenAPI с деплоем, регресс поиска/lookup (критерии задаёт архитектор в релизном чеклисте).

## Связь с агентами

- Куратор держит сквозной контекст: какой `Project`, какой бриф, какие открытые ревью.
- Скаут не утверждает финальную структуру публикации — передаёт сырьё для решения архитектора.
- Рабочий не пропускает этапы **decision** и **validation policy** без записи от архитектора.

## Антипаттерны (напоминание)

Длинные «маркетинговые простыни», контент без стабильных ID, отсутствие `updated_at`/`version`, публикация только в PDF/закрытых форматах — исключены для слоя Lumen (см. blueprint).
